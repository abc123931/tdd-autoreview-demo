# レビュー指示（Bugbot）

このファイルはコードレビュー専用の基準（Bugbot が読む正本）。
レビューは助言であり、マージはブロックしない（ブロックは CI のみ）。
関連: 厳格TDD=@docs/TDD.md / ロール分離=@docs/ROLES.md。

レビュアー（Bugbot）は実装者と**別血筋のモデル**であることが前提（ロール分離）。
実装者自身の自己採点ではなく、diff を批判的に独立検証する立場でレビューする。

## Important（マージ前に直すべき）の定義

本番を壊す / データ越境 / ロールバック阻害のみを Important とする。
スタイル・命名・リファクタ提案は最大でも Nit。

## Nit 上限

1 レビュー最大 5 件。超過は要約に「他 N 件」とだけ記す。
Important が無ければ要約を「ブロッキングな問題なし」で開始する。

## 報告しない（CI / Biome が見る）

lint / format / import 順 / 型エラー / 生成物 / lockfile。

## 常にチェック（TDD 規律 — 最重要）

### テスト同梱・Iron Law

- NG: 振る舞いを変える変更に対応する `*.test.ts` が無い
- OK: 変更と同じ PR にテストがある（RED を経た形跡＝境界値/異常系を含む）

### テストの健全性（実装従属の禁止）

- NG: レビュー/実装を通すために既存テストを**削除・緩和**している
  （アサーション弱体化・ケース削除・期待値を実装に合わせて書き換え）
- OK: 仕様変更が正当で、その理由が PR に説明され、テストが先に更新されている

### テスト設計技法

- NG: 複数条件分岐なのに一部の組み合わせしかテストしていない
- OK: デシジョンテーブルで全組み合わせ網羅（src/featureAccess.ts 参照）。
  数値境界は境界±1 を突く（src/tokenLimit.ts 参照）

### As-Is（characterization）の尊重

- NG: As-Is テスト（現挙動の固定）を「期待値がおかしい」と誤って Important 指摘する
- OK: characterization テストは現挙動の記録と理解した上でレビューする
  （src/legacyTaxLabel.ts / .test.ts）

## 常にチェック（ドメイン例）

### マルチテナンシー（最重要）

- NG: workspaceId で絞らない検索 / クエリ → 他テナント混入
- OK: 検索・取得は必ず workspaceId で絞る（src/multiTenant.ts 参照）

### 課金・上限判定

- NG: plan を見ずに上限を判定 / マジックナンバー直書き
- OK: resolvePlanLimit(plan) を必ず経由（src/tokenLimit.ts 参照）

## 検証バー

動作に関する指摘は file:line の根拠を必須とする（推論だけで指摘しない）。

## 再レビュー収束

2 回目以降は新規 Nit を抑制し Important のみ投稿する。

## 人間にエスカレーション（自動承認しない）

不可逆領域（DB マイグレーション・課金・認証・本番昇格）は人間が最終判断。
