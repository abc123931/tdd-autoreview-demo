# 厳格TDDの規律（このリポジトリの掟）

このプロジェクトは **厳格な TDD** を開発の前提とする。テストの網羅性と健全性を
もって「本番に出してよい」という信頼を担保し、人手レビューを最小化する。
AI エージェントに実装させる場合も、人間が実装する場合も、**例外なく本書に従う**。

---

## 1. Iron Law（鉄の掟）

> **失敗するテスト（RED）を書く前に、プロダクションコードを 1 行も書いてはならない。**

- 新しい振る舞いを足す／変える時は、必ず **先に失敗するテスト**を書く。
- バグ修正は **まず再現する失敗テスト**を書いてから直す（リグレッション固定）。
- 「テストなしの実装」「あとからテスト」は規律違反として扱う。

### Robert C. Martin の TDD 3 原則（運用形）

1. 失敗するテストがない限り、プロダクションコードを書かない。
2. コンパイル/型が通らない・失敗する以上にテストを書かない（書きすぎない）。
3. 現在失敗しているテストを通す以上に、プロダクションコードを書かない。

---

## 2. RED → GREEN → REFACTOR サイクル

1. **RED**: 失敗するテストを書く。**必ず一度赤くなるのを目視で確認**する
   （テストが何もせず緑なら、そのテストは無効）。
2. **GREEN**: テストを通す **最小限の** コードを書く（きれいさは後回し）。
3. **REFACTOR**: 緑を保ったまま設計を整える（重複除去・命名・抽出）。

各ステップは小さなコミットで残す（後述「退化」の証跡にもなる）。

```
test (RED) ──▶ commit "red: ..."
impl (GREEN) ─▶ commit "green: ..."
refactor ─────▶ commit "refactor: ..."
```

---

## 3. 退化（Degeneracy）— fake it → 三角測量 → 一般化

最初から「本物の実装」を書かない。**退化的実装**から始め、テストを増やして
一般化へ追い込む。これにより「テストが実装を駆動している」ことを担保する。

### 段階

1. **Fake it（定数返し）**: まず固定値を返してテストを通す。
2. **Triangulation（三角測量）**: 2 つ目以降の例を足し、固定値では通らなくして
   実装を一般化せざるを得なくする。
3. **Obvious Implementation（自明な実装）**: 自明な場合のみ直接書いてよい。

### 実演（`src/tokenLimit.ts` はこの順で作られた）

```ts
// RED 1: free の上限ちょうどは true
isWithinLimit("free", 300_000) // → 期待 true

// GREEN 1 (fake it): とりあえず true を返す
export const isWithinLimit = () => true;

// RED 2: free の超過は false（fake it では通らない＝三角測量）
isWithinLimit("free", 300_001) // → 期待 false

// GREEN 2 (一般化): 上限と比較する実装へ追い込まれる
export const isWithinLimit = (plan, used) => used <= resolvePlanLimit(plan);

// RED 3: 負値はエラー（異常系）→ GREEN: ガード追加 → REFACTOR
```

完全な段階は `docs/WALKTHROUGH.md`（コミット履歴）で追える。

## 4. テスト設計技法（何をテストするかの体系）

「思いつき」でケースを書かない。次の技法で**意図的に**ケースを選ぶ。

### 4.1 同値分割（Equivalence Partitioning）

入力を「同じ振る舞いになるグループ」に分け、各グループから代表を 1 つ選ぶ。

- 例（`src/tokenLimit.ts`）: 「上限以下」「上限超過」「不正(負値)」の 3 クラス。

### 4.2 境界値分析（Boundary Value Analysis）

バグは境界に潜む。**境界そのもの・境界±1** を必ず突く。

- 例: 上限 `300_000`（ちょうど＝OK）/ `300_001`（+1＝NG）。

### 4.3 デシジョンテーブル（Decision Table）

複数条件の組み合わせで結果が決まるロジックは、**全組み合わせを表にして網羅**する。

- 例（`src/featureAccess.ts`）: `isPaidPlan` × `withinTrial` × `hasPaidContract`
  の 2³=8 通りを表で定義し、テストで全行を検証する。

| isPaidPlan | withinTrial | hasPaidContract | canUseFreee |
|---|---|---|---|
| true | * | * | true |
| false | true | false | true |
| false | true | true | false |
| false | false | * | false |

## 5. As-Is テスト（Characterization / ゴールデンマスター）

仕様書のない既存（レガシー）コードに触れる時は、**今の挙動をそのまま固定する
テスト**を先に書いてから変更する。

- 目的: リファクタで**現挙動を壊していない**ことを保証する安全網。
- 注意: As-Is テストは「正しい仕様」ではなく「**現在の事実**」を写し取る。
  仕様の良し悪しの判断は別問題（テストは現挙動を記録するだけ）。
- 例: `src/legacyTaxLabel.ts` の挙動（8% 切り捨て・負値もそのまま）を
  `legacyTaxLabel.test.ts` で固定。安全になってから初めてリファクタ可能。

## 6. ミューテーションテスト（テストの健全性を測る）

カバレッジ 100% でも「アサーションが甘い」テストはバグを見逃す。
**ミューテーション（コードを意図的に改変）してテストが失敗するか**を測り、
テスト自身の品質を担保する。

- ツール: Stryker（`npm run test:mutation`）。
- 指標: **Mutation Score**（殺したミュータント / 全ミュータント）。
- 生き残ったミュータント = テストが検出できない変更 = **テストの穴**。
  穴を塞ぐテストを追加する（これも RED から）。
- 運用: PR の通常 CI とは別の重いジョブ（`.github/workflows/mutation.yml`）で実行。

---

## 7. 何をテストし、何をしないか

- **手厚く**: ロジック・分岐・境界値・アクセス制御・**金銭/権限に直結する箇所**。
- **純粋関数に切り出してテスト**: Slack/DB/外部 API などの副作用は薄い殻に追い出し、
  中核ロジックを純粋関数化してユニットtestする。
- CI が見るもの（lint/format/型）はテストの対象にしない。

---

## 8. チェックリスト（PR を出す前）

- [ ] 変更に対応する **失敗テストを先に書いた**（RED を目視した）
- [ ] 最小実装で GREEN にした → REFACTOR した
- [ ] 同値分割・境界値（必要ならデシジョンテーブル）で**意図的に**ケース選定した
- [ ] レガシーに触れたなら As-Is テストで現挙動を固定した
- [ ] `npm run test:run` / `lint` / `typecheck` が緑
- [ ] （重要変更なら）`npm run test:mutation` で生存ミュータントを確認した
