# ロール分離と「別血筋モデル」

自動レビュー／AI 駆動開発の健全性は、**実装した主体に自分自身を採点させない**
ことで担保される。本書はその役割分担を定義する。

## なぜ分離するのか

- 実装者は自分のコードに「投資」しているため、欠陥を見落としやすい。
- 同じモデル・同じ文脈で「書いて・テストして・レビューする」と、
  同じ盲点がすべての工程に伝播する（バイアスの共有）。
- 出典の思想:
  - Claude security-guidance:「コードを書いた同じ Claude に採点させない。
    レビュアーは新しい文脈・別呼び出しで diff から始める」
  - ZOZO: Reviewer Pass と Validator Pass を**別ペルソナ**に分離して誤検知抑制。
  - カウシェ: 3 ペルソナ並列レビュー＋計測と改善を**別エージェント**に分離
    （自分の改善効果を自分で計測させない）。

## 3 つのロール（別血筋＝異なるモデル系統で担う）

| ロール | 責務 | 担い手の原則 |
|---|---|---|
| **Implementer（実装）** | GREEN にする最小実装・REFACTOR | モデル A（または人間） |
| **Test Author（テスト設計）** | RED テストの設計（境界値/同値/デシジョン表/As-Is） | **実装と別の血筋**（モデル B か人間）が望ましい |
| **Evaluator（レビュー）** | PR の diff を批判的に検証し指摘 | **Bugbot（モデル C）**＝実装者と必ず別 |

「血筋（lineage）が別」とは、**異なるモデル系統**（例: 実装は Cursor の Composer、
レビューは Bugbot、テスト設計はさらに別セッション/別モデル）を使うこと。
最低限の不変条件は **Implementer ≠ Evaluator**。

## 最低限守る不変条件

1. **Implementer ≠ Evaluator**: 実装した本人（同一モデル・同一セッション）に
   最終レビューをさせない。レビューは Bugbot（別血筋）に委ねる。
2. **テストは実装に追従して甘くしない**: 実装を通すためにテストを緩める変更は
   レビューで Important として扱う（テストが実装の従属物になると無意味）。
3. **計測と改善の分離**: レビュー精度の計測（誤検知/見逃し集計）と、ルール改善は
   別の工程・別の担当で行う。

## 実運用での割り当て（このリポジトリ）

```
[RED] Test Author（別セッション/別モデル推奨）
   └─ 失敗テストを設計（docs/TDD.md の技法に従う）
        ↓
[GREEN/REFACTOR] Implementer（モデルA or 人間）
   └─ 最小実装→リファクタ
        ↓
[PR] CI（決定論・ブロッカー） + Evaluator=Bugbot（別血筋・助言）
   └─ .cursor/BUGBOT.md 基準でレビュー
        ↓
[昇格] 人間（不可逆領域の最終判断）
```

## アンチパターン

- 実装したエージェントにそのまま「自分でレビューして」と頼む（血筋が同じ）。
- レビューで落ちたから**テストを消して/緩めて**通す。
- 1 つのモデル・1 セッションで RED もテストも実装もレビューも完結させる。
